# Phase-wise Migration Plan: dicomweb-api

## Current Architecture Summary

- **Framework:** Node.js + Fastify
- **DB:** SQLite (`image.db`) — managed internally by `dicom-dimse-native` (C++ binding), **no direct SQL in JS code**
- **Import:** `storescu.js` reads files from `./import/`, sends via DIMSE C-STORE to local PACS server (port 8888), which writes to `./data/{StudyUID}/` and `image.db`
- **Queries:** All QIDO queries go through `dimse.findScu()` → local PACS → SQLite
- **File serving:** WADO reads directly from `./data/` filesystem

> **Critical constraint:** SQLite is managed by the C++ `dicom-dimse-native` library, not by JS code. You cannot simply swap a DB driver — the entire data layer needs to be re-architected.

---

## Phase 1 — PostgreSQL Migration (Replace SQLite metadata layer)

**Goal:** Decouple metadata storage from the DIMSE native library and store it in PostgreSQL.

### 1.1 — Schema Design
Define PostgreSQL tables mirroring DICOM hierarchy:
```
patients → studies → series → instances
```
Each table stores relevant DICOM tags (PatientID, StudyDate, Modality, SOPInstanceUID, file path, etc.).

### 1.2 — Add PostgreSQL Client
- Add `pg` (node-postgres) or `knex` as the query builder
- Add `pg-migrate` or `knex` migrations for schema versioning
- Add DB config (`host`, `port`, `database`, `user`, `password`) to `config/default.js` and environment-specific overrides

### 1.3 — Create a Metadata Service Layer
Create `src/db/` module:
- `src/db/connection.js` — pool setup
- `src/db/migrations/` — schema migration files
- `src/db/studyRepository.js` — CRUD for studies/series/instances

### 1.4 — Modify Import to Write Metadata to PostgreSQL
In `storescu.js`, after `dimse.storeScu()` succeeds:
- Parse each DICOM file using `dicom-parser` (already a dependency)
- Extract: PatientID, StudyInstanceUID, SeriesInstanceUID, SOPInstanceUID, Modality, StudyDate, etc.
- Upsert into PostgreSQL tables

### 1.5 — Replace QIDO Queries (routes.js)
Replace `utils.doFind()` (which queries DIMSE → SQLite) with direct PostgreSQL queries from the repository layer:
- `GET /rs/studies` → `SELECT` from `studies` table with filters
- `GET /rs/studies/:uid/series` → `SELECT` from `series` table
- `GET /rs/studies/:uid/series/:uid/instances` → `SELECT` from `instances` table

The DIMSE SCP (`startScp`) can remain running for DICOM network reception, but QIDO queries bypass it.

### 1.6 — Data Migration Script
Write a one-time migration script (`src/scripts/migrate-sqlite-to-pg.js`):
- Walk `./data/` filesystem
- Parse each DICOM file to extract metadata
- Bulk insert into PostgreSQL

### 1.7 — Testing
- Verify all QIDO-RS responses match original output
- Verify WADO-RS pixel data still served correctly
- Test duplicate handling during import

## Phase 2 — Cloud Migration (S3 + Lambda replaces local folders)

**Goal:** Replace `./import/` and `./data/` local folders with S3 buckets; replace `npm run import` with event-driven Lambda processing.

### 2.1 — S3 Bucket Setup (Infrastructure)
Create two S3 buckets:

| Bucket | Purpose |
|--------|---------|
| `dicomweb-import` | Drop zone — DICOM files uploaded here trigger processing |
| `dicomweb-data` | Permanent storage — processed DICOM files stored here |

Configure S3 bucket policies, versioning, and lifecycle rules as needed.

### 2.2 — Lambda Function Design
Create a Lambda function (`dicomweb-import-processor`):

**Trigger:** S3 `ObjectCreated` event on `dicomweb-import` bucket

**Lambda logic (replaces `storescu.js`):**
1. Receive S3 event → get bucket + object key
2. Download DICOM file from `dicomweb-import` S3
3. Parse DICOM metadata (using a Lambda layer with `dicom-parser`)
4. Extract: PatientID, StudyUID, SeriesUID, SOPUID, modality, dates
5. Copy/move file to `dicomweb-data` bucket under `{StudyUID}/{SOPInstanceUID}` key path
6. Upsert metadata into PostgreSQL RDS (from Phase 1)
7. Delete file from `dicomweb-import` bucket (or archive it)

**Lambda packaging:**
- Node.js runtime
- Bundle `dicom-parser` and `pg` client
- Store DB credentials in AWS Secrets Manager or Parameter Store

### 2.3 — PostgreSQL on AWS RDS
- Migrate the PostgreSQL instance (from Phase 1) to **Amazon RDS for PostgreSQL**
- Configure VPC, security groups so Lambda can reach RDS
- Update API server DB connection config to point to RDS endpoint

### 2.4 — Modify API Server to Serve Files from S3
Update `routes.js` WADO endpoints:
- `GET .../frames/:frame` — instead of reading from `./data/`, generate a **pre-signed S3 URL** from `dicomweb-data` and redirect, **or** stream the object directly
- `GET /wadouri` — same S3 streaming approach
- Remove `storagePath` local filesystem dependency

Add AWS SDK (`@aws-sdk/client-s3`) to the API project.

### 2.5 — Remove Local Import Command
- Retire `storescu.js` and `npm run import` script
- Update `README` and `package.json` scripts
- Upload workflow: users upload DICOM files directly to `dicomweb-import` S3 bucket (via console, CLI, or a future upload API endpoint)

### 2.6 — Infrastructure as Code
Write infrastructure definitions (AWS CDK or Terraform) for:
- S3 buckets with event notifications
- Lambda function + IAM role (S3 read/write + RDS access + Secrets Manager)
- RDS PostgreSQL instance
- VPC, subnets, security groups
- (Optional) API Gateway if the Fastify API itself moves to Lambda/ECS

## Key Risks & Notes

| Risk | Mitigation |
|------|-----------|
| `dicom-dimse-native` is a C++ binary with no PostgreSQL support | Bypass it for DB queries entirely; keep it only for DICOM protocol (SCP server) or replace it later |
| Lambda has a 15-min timeout and 10GB memory limit | Large DICOM studies (e.g., CT with 1000 slices) should upload per-file, not as a zip; or use Step Functions for batch processing |
| DICOM transfer syntax conversion (`recompress()`) is CPU-intensive | For cloud, offload recompression to a separate Lambda or ECS task |
| Cold-start latency on Lambda | Use provisioned concurrency if import throughput is high |
| RDS must be in same VPC as Lambda | Plan VPC architecture before Phase 2 begins |
