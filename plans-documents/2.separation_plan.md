# Plan: Separate API Server from OHIF Viewer

## Current Coupling Points

Before listing steps, here is exactly where and how the two are coupled today:

| Location | Line(s) | What it does |
|----------|---------|--------------|
| `src/app.js` | 17вҖ“19 | Registers `@fastify/static` to serve `../public` (the OHIF viewer) |
| `src/app.js` | 20вҖ“22 | `setNotFoundHandler` returns `index.html` (SPA client-side routing fallback) |
| `src/app.js` | 35вҖ“38 | Routes registered a second time under `/viewer` prefix, for OHIF deep links |
| `src/app.js` | 25вҖ“30 | Helmet CORP/COOP/COEP headers set here, but they are needed by the viewer (for WASM/MPR), not the API |
| `public/app-config.js` | 63вҖ“65 | `qidoRoot: "rs"`, `wadoRoot: "rs"`, `wadoUriRoot: "wadouri"` are **relative paths** вҖ” assumes viewer is served from the same origin as the API |

## Step-by-Step Actions

---

### Step 1 вҖ” Create a separate entry point for the viewer server

**File to create:** `viewer/server.js`

Create a minimal Fastify (or http-server) static file server that:
- Serves the `public/` directory
- Has `setNotFoundHandler` returning `index.html` (for SPA routing)
- Sets the CORP/COOP/COEP security headers required for WASM (MPR feature)
- Listens on a configurable `viewerPort` (e.g. 3000)

These responsibilities are currently mixed into `src/app.js` lines 17вҖ“22 and 25вҖ“30.

---

### Step 2 вҖ” Remove viewer responsibilities from `src/app.js`

**File to edit:** `src/app.js`

Remove the following from the API server:

1. **Lines 17вҖ“19** вҖ” Remove `@fastify/static` registration (no longer serving `public/`)
2. **Lines 20вҖ“22** вҖ” Remove `setNotFoundHandler` (SPA fallback belongs in viewer server)
3. **Lines 35вҖ“38** вҖ” Remove duplicate route registration with `/viewer` prefix (only needed because OHIF deep-links resolved against the same origin)
4. **Lines 25вҖ“30** вҖ” Remove or simplify Helmet CORP/COOP/COEP headers. The API server only needs standard CORS; the viewer server needs the WASM headers.
5. **Remove** `@fastify/static` from the `require` imports at the top

---

### Step 3 вҖ” Update CORS configuration on the API server

**File to edit:** `src/app.js`

The current CORS config is `{}` (allow all origins). When viewer runs on a different origin, it will still work, but it should be tightened to only allow the viewer's origin:

```js
server.register(fastifyCors, {
  origin: config.get('viewerOrigin'), // e.g. "http://localhost:3000"
});
```

Add `viewerOrigin` to config (see Step 5).

---

### Step 4 вҖ” Update `public/app-config.js` to use absolute API URLs

**File to edit:** `public/app-config.js`

The viewer currently uses **relative paths** for API roots:
```js
wadoUriRoot: "wadouri",
qidoRoot: "rs",
wadoRoot: "rs",
```

These must become **absolute URLs** pointing to the API server:
```js
wadoUriRoot: "http://localhost:5001/wadouri",
qidoRoot: "http://localhost:5001/rs",
wadoRoot: "http://localhost:5001/rs",
```

Because `app-config.js` is a plain JS file (not a module), the API server URL should either be:
- Hardcoded for a fixed deployment, or
- Injected at build/serve time via a template substitution in the viewer server

---

### Step 5 вҖ” Add new configuration keys

**File to edit:** `config/default.js`

Add:
```js
viewerPort: 3000,               // Port for the viewer static server
viewerOrigin: "http://localhost:3000",  // Used by API server for CORS
apiPort: 5001,                  // Rename from webserverPort for clarity (or keep both)
```

Optionally rename `webserverPort` вҶ’ `apiPort` and update all references in `src/app.js`.

---

### Step 6 вҖ” Update `package.json` scripts

**File to edit:** `package.json`

Replace the single `start` script with separate scripts:
```json
"scripts": {
  "start": "node src/app.js & node viewer/server.js",
  "start:api": "node src/app.js",
  "start:viewer": "node viewer/server.js",
  "import": "node src/storescu.js"
}
```

This lets each application start independently:
- `npm run start:api` вҖ” starts only the DICOM API + DIMSE server
- `npm run start:viewer` вҖ” starts only the static viewer server
- `npm start` вҖ” starts both together (for convenience)

---

### Step 7 вҖ” Remove `@fastify/static` from API server dependencies (optional)

**File to edit:** `package.json`

If `@fastify/static` is no longer used by `src/app.js` and is only used by the new `viewer/server.js`, no package change is needed (it is reused). But if the viewer is moved to its own package in the future, this would move with it.

No immediate action required unless doing a full monorepo split.

---

### Step 8 вҖ” Handle `app-config.js` API URL dynamically (optional but recommended)

**File to create:** `viewer/server.js` enhancement

Since `app-config.js` is a static JS file, making the API URL configurable without editing the file directly requires the viewer server to serve a dynamically generated `app-config.js`:

```js
server.get('/app-config.js', (req, reply) => {
  const apiUrl = config.get('apiUrl'); // e.g. http://localhost:5001
  reply.header('Content-Type', 'application/javascript');
  reply.send(`window.config = { ..., qidoRoot: "${apiUrl}/rs", wadoRoot: "${apiUrl}/rs", ... }`);
});
```

This avoids hardcoding the API URL inside the viewer's static assets.

## Risk / Notes

- **CORS** вҖ” The biggest functional change. The API must explicitly allow the viewer's origin or requests will be blocked by the browser.
- **WASM headers (CORP/COOP/COEP)** вҖ” These must be set on the viewer server. If they are removed from the API server without being added to the viewer server, MPR (3D rendering) will break.
- **`/viewer` prefix routes** вҖ” Currently OHIF uses both `/` and `/viewer` as base paths. After separation, only `/` is needed on the viewer server. The API server no longer needs the `/viewer` prefix.
- **`app-config.js` URL** вҖ” If the API server URL is not configurable (Step 8), the file must be manually edited for each deployment environment.
